# Post-build PR and Close Flow

## Background and Design Rationale

Today `ship` is overloaded: it folds, sets `status: done`, moves the spec to
`_done/`, regenerates the committed `INDEX.md`, and hands off to `mochiflow pr` —
all in one close-out commit created *before* the PR is opened. Because lifecycle
state is expressed by *writing* a terminal state into tracked files, and the base
branch cannot be committed to directly, the terminal write is forced to happen
pre-merge. PR feedback then has to resurrect completed work (`_done/` → active,
`done` → `approved`). This is backwards for a still-open review.

Agreed redesign (decided in discuss; see `pitch.md`):

- Replace `ship` entirely with three state-driven actions: `open`, `update`,
  `close`.
- Separate **asserted** spec states (stored in `spec.yaml`: `draft → approved →
  accepted`) from **derived** delivery facts (`in_review`, `merged`) that are
  observed from VCS/provider and never stored. `done` stops being a stored state.
- Stop moving specs to `_done/`; a spec lives flat at `{specs_dir}/{slug}/` for
  its whole life. Existing `_done/` archives stay read-only.
- The kanban is computed, not committed: `mochiflow status` renders it on demand,
  and `INDEX.md` becomes a gitignored generated artifact regenerated at the end
  of mochiflow commands. No git hooks.
- The fold lands in the `open` PR (reviewed, merged atomically); `close` is local
  hygiene only and writes nothing to the base branch.

Rejected alternatives are recorded in `pitch.md ## Alternatives Considered`
(smarter `ship`; archive-before-PR; post-merge base commit at close; follow-up
archive PR; optimistic in-PR archive; symlink projection board; committed live
`INDEX.md`; git hooks).

## User Story

As a MochiFlow user delivering a built spec, I want the post-build flow to offer
the next safe action based on real repository state — prepare and open a PR,
address review feedback, and close after merge — so that work is never marked
complete or archived before it actually merges, and feedback never requires
resurrecting archived work.

## Scope

- In:
  - Remove `ship` as a user-facing verb/trigger; add `open` / `update` / `close`
    engine command procedures.
  - Introduce the `accepted` asserted state; stop writing `done` and stop the
    `_done/` archive move for new specs.
  - Derive `in_review` / `merged` from provider/git; never store them.
  - New `mochiflow status` board command; make `INDEX.md` a gitignored generated
    artifact auto-regenerated by mochiflow commands.
  - Adjust `mochiflow pr` pre-flight; repurpose the deterministic ship close-out
    into an `accepted`-setting close-out with no `done`/`_done`/`INDEX` write.
  - Stale-base guard at spec start (fetch + branch from `origin/{base_branch}` +
    warn when local base is behind).
  - Rewrite affected engine docs (router, build, discuss, workflow, git, command
    docs) and adapter templates; keep `doctor`/conformance consistent.
  - Backward compatibility: existing `_done/` specs (status `done`) remain
    read-only and render as merged.
- Out:
  - Release automation (`release` stays a separate future feature).
  - New PR providers beyond the current `github` built-in / `pr_driver` / manual
    handoff (gitlab/azure remain future work).
  - Any base-branch commit/push, git hooks, symlink board, or committed live
    kanban file.
  - Rewriting or migrating existing archived `_done/` specs.
  - Automated `join`/`upgrade` migration to untrack an already-committed
    `INDEX.md` in existing repos (done once, manually).
  - Designing or redefining a no-PR fast path flow. It is not part of this
    PR-centric redesign and is not redefined here; it inherits the global
    invariants (sets at most `accepted`, never `done`, no `_done/` move, no
    committed `INDEX.md`, no fold) per AC-05 / AC-12.

## Edge Cases

- `provider = none`: derivation must work with local git only and must not error.
- No remote / offline: `status` uses last-known local state; `--fetch` is the
  network path. Derivation never hard-fails the command.
- Squash merge: `merged` derivation relies on the `Spec: {slug}` trailer
  surviving into the base branch; if it is absent and no provider is available,
  the spec stays In Review until the trailer/provider confirms the merge — the
  human merge report still initiates `close` locally but persists no merged flag.
- Detached HEAD or a non-spec branch: `status` and the stale-base guard degrade
  gracefully (no crash; clear message).
- A user manually edits or deletes the gitignored `INDEX.md`: the next
  state-changing command or `mochiflow index` regenerates it (`mochiflow status`
  does not); it is never staged or committed.
- Local base branch behind origin when starting a new spec: warn and branch from
  `origin/{base_branch}`, never from a stale local base.
- A spec whose delivery PR is merged but `close` was never run: `status` still
  shows it as Done (derived), independent of `close`.
- Conflicting delivery signals (e.g. an open PR plus a `Spec: {slug}` trailer
  already reachable from base): resolved by precedence — Done outranks In Review.

## Acceptance Criteria (EARS)

- AC-01: THE SYSTEM SHALL expose `open`, `update`, and `close` as engine command
  procedures and SHALL NOT expose `ship` as a user-facing verb, trigger, or
  trigger pattern in any engine doc, router, or adapter template.
- AC-02: WHEN build completes, THE SYSTEM SHALL leave `spec.yaml` `status` at
  `approved` and SHALL NOT create a PR, set a terminal state, or move the spec
  directory.
- AC-03: THE SYSTEM SHALL accept `accepted` as a valid `spec.yaml` `status`
  across the contract surface — `contracts/spec.schema.json` enum, the contracts
  version gate / `contracts/contracts.lock`, `lint.rs` allowed statuses, and the
  conformance fixtures — for the flow `draft → approved → accepted`, and
  `mochiflow lint` SHALL pass for an `accepted` spec whose acceptance conditions
  hold.
- AC-04: WHERE `risk` is `elevated` or `critical`, THE SYSTEM SHALL require a
  recorded reviewer verdict (`pass` / `pass-with-comments`) before `status`
  becomes `accepted`.
- AC-05: THE SYSTEM SHALL NOT write `status: done` for any spec and SHALL NOT
  move any spec into `_done/` (a global invariant that also binds any no-PR fast
  path).
- AC-06: THE SYSTEM SHALL treat existing `{specs_dir}/_done/` specs as read-only,
  SHALL keep `mochiflow lint` passing for their stored `done` status, SHALL render
  them in the Done view, and SHALL reject `status: done` on an active
  (non-`_done/`) spec.
- AC-07: THE SYSTEM SHALL derive `merged` for a spec using, in priority order,
  the provider API when configured and available, else a `Spec: {slug}` trailer
  reachable from `origin/{base_branch}` (two signals only). The human merge
  report SHALL only trigger `close` locally and SHALL NOT be persisted as a
  merged signal.
- AC-08: WHEN deriving delivery state (including `provider = none`), THE SYSTEM
  SHALL resolve each spec to exactly one column using the precedence Done > In
  Review > Ready > Active > Backlog: Done when the provider reports merged or a
  `Spec: {slug}` trailer is reachable from `origin/{base_branch}`; In Review when
  not Done and either the provider reports an open PR or (`provider = none`) the
  spec branch is pushed to `origin` and unmerged; Ready when `accepted` but
  neither Done nor In Review (e.g. accepted-unpushed); Active when `draft`/
  `approved`; Backlog when the entry is a `_backlog/` seed (not yet a spec). WHEN
  the provider is unavailable, THE SYSTEM SHALL fall back to the
  local-git signals and SHALL NOT fail the command.
- AC-09: THE SYSTEM SHALL provide `mochiflow status` that prints a board with the
  columns Backlog, Active, Ready, In Review, and Done, computed from `spec.yaml`
  asserted state plus derived delivery state, and SHALL be read-only (it writes
  no file, including `INDEX.md`).
- AC-10: WHEN `mochiflow status --fetch` is invoked, THE SYSTEM SHALL fetch
  `origin` before computing derived state.
- AC-11: WHEN a state-changing mochiflow command (`accept`, `pr`, `index`, and
  the discuss/plan/build/open/update/close procedures' commit steps) completes,
  THE SYSTEM SHALL regenerate `INDEX.md` from the same board computation used by
  `status`; `mochiflow status` SHALL NOT regenerate or write `INDEX.md`.
- AC-12: THE SYSTEM SHALL keep `INDEX.md` gitignored and SHALL NOT stage or commit
  `INDEX.md` in any command, including the `open` close-out commit.
- AC-13: WHEN `open` runs, THE SYSTEM SHALL perform, in order, (a) acceptance
  including the human QA round-trip, (b) finalize the fold by writing the durable
  ADR knowledge, (c) run `accept` to set `accepted` and create the single
  feature-branch close-out commit that includes the spec and the already-written
  ADR fold, (d) generate the PR title/body, (e) require the approve-PR human
  gate, (f) push and create the PR; and SHALL NOT create the PR before the
  approve-PR gate.
- AC-14: WHEN `update` runs, THE SYSTEM SHALL apply code changes through the
  `build` loop, re-verify, push, and update PR metadata, and SHALL NOT move the
  spec directory or revert any asserted state.
- AC-15: WHEN `close` runs after a confirmed merge, THE SYSTEM SHALL perform only
  local hygiene (switch to base, fast-forward pull, delete the local branch,
  clear `{install_dir}/state/{slug}/`, regenerate the board) and SHALL NOT commit
  or push to the base branch.
- AC-16: WHEN a new spec is started (discuss branch creation), THE SYSTEM SHALL
  create the branch from `origin/{base_branch}` and SHALL warn when the local
  base branch is behind `origin/{base_branch}`.
- AC-17: WHEN `mochiflow pr` runs its pre-flight, THE SYSTEM SHALL require the
  spec to be committed at `{specs_dir}/{slug}/` with status `accepted` and a
  `Spec: {slug}` trailer present, and SHALL NOT require the spec under `_done/`
  or a `done` status.
- AC-18: THE SYSTEM SHALL keep `mochiflow doctor` and the conformance test suite
  green with the new command set (`status` and `accept` present, `ship` absent)
  and the new verb vocabulary (`open` / `update` / `close`).

## Completion Conditions

- Every AC appears in the AC Verification Matrix (below) with a
  done-eligible result token (`PASS`, `CONFIRMED`, or `N/A: <reason>`).
- The CLI `default` verification command passes; conformance and command tests
  cover the new/changed behavior.
- A reviewer verdict is recorded in `design.md ## Review Results` (risk is
  `elevated`).
- The full `cargo test` suite is green. The specific test names called out in
  tasks are representative, not exhaustive: any remaining test, fixture, or
  generated artifact still coupled to the removed `ship` / `done` / `_done/` /
  committed-`INDEX` vocabulary is updated within the task that changes that
  behavior, and the green suite (plus the `rg` sweep in T-009) is the backstop.
