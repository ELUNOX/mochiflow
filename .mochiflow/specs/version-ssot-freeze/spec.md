# Version SSOT + freeze command

## Background and Design Rationale

**Problem.** mochiflow's release version lives in three hand-maintained places
that drift independently:

- `cli/Cargo.toml` `[workspace.package].version` (currently `1.1.3`) вҖ” drives
  `--version` via clap `#[command(version)]` (`CARGO_PKG_VERSION`).
- `engine/VERSION` (currently `1.1.3`) вҖ” read at build time by
  `bundled_engine_version()` (`main.rs`) and vendored into user projects; also
  the `version` field written into `engine/MANIFEST.json` by
  `write_manifest_for_engine_dir()` (`upgrade.rs`).
- `contracts/contracts.lock` `version` (currently `1.1.3`) вҖ” the frozen-surface
  stamp checked by `version_gate_consistent` (`conformance.rs`).

A real three-way drift incident occurred on 2026-06-21 (engine/VERSION=1.1.3,
Cargo.toml=1.1.2, contracts.lock.version=2.0.0 in one working tree).
`release.md` Step 1 explicitly documents this as "Known drift to watch for".

**Secondary problem вҖ” manual hash management.** Two derived integrity facts must
be regenerated by hand or via scattered commands:

- `engine/MANIFEST.json` вҖ” the full engine file-hash map, regenerated by the
  hidden `mochiflow engine manifest` subcommand or as a side effect of
  `upgrade`. Editing any engine file (e.g. an adapter template) without
  regenerating it makes `doctor engine` fail with MANIFEST drift. This was hit
  in this very session when the kiro adapter templates were edited.
- `contracts/contracts.lock` вҖ” the `{version, hash}` frozen-surface lock. Its
  hash is computed by `compute_contracts_hash()` which lives **only in the test
  file** `conformance.rs`, so there is no production command to regenerate the
  lock; contributors edit it by hand or copy the hash out of a failing test.

**Chosen approach.** Make `cli/Cargo.toml` `[workspace.package].version` the
single source of truth and add one `mochiflow freeze` command that regenerates
all derived files in one shot, with `freeze --check` for CI enforcement.
`compute_contracts_hash()` moves into production library code so the conformance
test and `freeze` share one implementation.

**Rejected alternatives.**

- *Keep `engine/VERSION` as a separate hand-edited source.* Rejected: it caused
  the real three-way drift above.
- *Keep `mochiflow engine manifest` as a separate command alongside freeze.*
  Rejected: two regeneration commands create "which one do I call?" ambiguity.
  `freeze` is idempotent and handles the superset, so `engine manifest` is
  absorbed.
- *Compute version from `env!("CARGO_PKG_VERSION")` inside freeze.* Rejected:
  freeze is a repo-tree integrity tool and must reflect the version declared in
  source (`cli/Cargo.toml`), not the version baked into whatever binary happens
  to run it.

**Clarification of the two integrity artifacts (not changed by this spec).**
`engine/MANIFEST.json` (full engine file-hash map, integrity/drift baseline) and
`contracts/contracts.lock` (schemas + golden hash, version gate) are
deliberately separate and stay separate. `freeze` writes both; it does not merge
them. The frozen-surface hash input remains exactly `contracts/*.json` (sorted)
followed by `tests/conformance/golden/**` (sorted), unchanged from
`compute_contracts_hash()` today.

## User Story

As a mochiflow maintainer or AI agent doing a release, I want a single version
source and one command that regenerates every derived file, so that I bump the
version in one place and never chase three-way version drift or hand-edit
integrity hashes.

## Scope

- In:
  - New `mochiflow freeze` command (write mode) and `mochiflow freeze --check`
    (read-only drift report, non-zero exit on staleness).
  - `freeze` reads `[workspace.package].version` from `cli/Cargo.toml` at runtime
    and writes the three derived files: `engine/VERSION`,
    `engine/MANIFEST.json`, `contracts/contracts.lock`.
  - Move frozen-surface hash logic from `conformance.rs`
    (`compute_contracts_hash`) into `mochiflow-core` as the single
    implementation; refactor the conformance test to call it.
  - Remove the hidden `mochiflow engine manifest` subcommand (absorbed by
    `freeze`).
  - Simplify the version gate to one rule and delete the drift-tolerating
    escape hatch (`|| engine_version != lock_version`).
  - Replace the literal `version_is_1_1_3` test with a dynamic test comparing
    `--version` to `env!("CARGO_PKG_VERSION")`.
  - Add a `mochiflow freeze --check` step to CI (`.github/workflows/ci.yml`).
  - Update contributor docs that describe the version/freeze procedure
    (`contracts/VERSIONING.md`, release steering, CONTRIBUTING.md, PR template)
    and remove the "Known drift to watch for" warning.
- Out:
  - No change to the frozen-surface hash input set (schemas + golden).
  - No change to `mochiflow doctor engine` drift-detection behavior (it keeps
    reading `engine/MANIFEST.json`); freeze only regenerates the file doctor
    reads.
  - No change to `schema_version` semantics or consumer `config.toml`.
  - No new user-facing runtime behavior beyond the `freeze` command itself.
  - No automatic version bumping (freeze stamps the current declared version; it
    does not decide semver increments).

## Edge Cases

- `cli/Cargo.toml` missing or unparseable `[workspace.package].version` вҶ’
  `freeze` exits non-zero with a clear error; writes nothing.
- `freeze --check` when a derived file is stale вҶ’ exits non-zero, naming which
  file(s) drifted; writes nothing.
- `freeze` run from a non-repo or installed-engine tree where
  `tests/conformance/golden/` or `contracts/` is absent вҶ’ it is a
  repo-maintenance tool; on a missing frozen surface it fails clearly rather
  than writing a partial/empty lock.
- `freeze` is idempotent: running it twice with no source changes produces no
  diff and `--check` passes.
- Engine prose-only edit (e.g. a template): `freeze` updates
  `engine/MANIFEST.json` but the `contracts.lock` hash is unchanged (engine
  files are not in the frozen surface), so the version gate does not trip.

## Acceptance Criteria (EARS)

- AC-01: THE SYSTEM SHALL treat `cli/Cargo.toml` `[workspace.package].version`
  as the sole version source, deriving `--version` output, `engine/VERSION`,
  `engine/MANIFEST.json` `version`, and `contracts/contracts.lock` `version`
  from it.
- AC-02: WHEN `mochiflow freeze` runs, THE SYSTEM SHALL write `engine/VERSION`,
  `engine/MANIFEST.json`, and `contracts/contracts.lock` consistent with the
  current `cli/Cargo.toml` workspace version and the current frozen-surface
  hash, in a single invocation.
- AC-03: WHEN `mochiflow freeze` reads the workspace version, THE SYSTEM SHALL
  read it from `cli/Cargo.toml` at runtime rather than from
  `env!("CARGO_PKG_VERSION")`.
- AC-04: WHEN `mochiflow freeze --check` runs AND any derived file
  (`engine/VERSION`, `engine/MANIFEST.json`, `contracts/contracts.lock`) is
  stale relative to the source, THE SYSTEM SHALL exit non-zero and name the
  stale file(s) without writing.
- AC-05: WHEN `mochiflow freeze` runs twice with no intervening source change,
  THE SYSTEM SHALL produce no file changes on the second run (idempotent) and
  `freeze --check` SHALL pass.
- AC-06: THE SYSTEM SHALL compute the frozen-surface hash from exactly one
  implementation in `mochiflow-core`, called by both `freeze` and the
  conformance version-gate test, over `contracts/*.json` (sorted) then
  `tests/conformance/golden/**` (sorted).
- AC-07: THE SYSTEM SHALL NOT expose a `mochiflow engine manifest` subcommand
  (it is removed and absorbed into `freeze`).
- AC-08: THE SYSTEM SHALL enforce the version gate as a single rule вҖ”
  `contracts.lock.hash == computed_hash` AND
  `contracts.lock.version == cli/Cargo.toml workspace version == engine/VERSION`
  вҖ” with no drift-tolerating escape clause.
- AC-09: WHERE the CLI version is asserted in tests, THE SYSTEM SHALL compare
  `--version` output against `env!("CARGO_PKG_VERSION")` dynamically, requiring
  no test edit on a version bump.
- AC-10: WHEN CI runs on a pull request, THE SYSTEM SHALL execute
  `mochiflow freeze --check` and fail the job if any derived file is stale.
- AC-11: THE SYSTEM SHALL update contributor documentation
  (`contracts/VERSIONING.md`, `docs/versioning.md`, `CONTRIBUTING.md`,
  `.kiro/steering/release.md`, `.github/PULL_REQUEST_TEMPLATE.md`) to
  describe the SSOT + `freeze` workflow and remove the "Known drift to watch
  for" warning.

## QA Scenarios

| QA | Scope | Steps | Expected result |
| --- | --- | --- | --- |
| QA-01 | cli | Run `cargo test --manifest-path cli/Cargo.toml` | All tests pass, including the refactored version-gate and dynamic version tests |
| QA-02 | cli | Bump `cli/Cargo.toml` workspace version, run `mochiflow freeze`, inspect `engine/VERSION`, `engine/MANIFEST.json` `version`, `contracts/contracts.lock` `version` | All three reflect the new version; `freeze --check` then passes |
| QA-03 | cli | With a clean tree, run `mochiflow freeze` then `git status` | No file changes (idempotent) |
| QA-04 | cli | Hand-edit `engine/VERSION` to a wrong value, run `mochiflow freeze --check` | Exits non-zero, names `engine/VERSION` as stale, writes nothing |
| QA-05 | cli | Run `mochiflow engine manifest` | Command no longer exists (clap error / unknown subcommand) |
| QA-06 | cli | Edit an engine template file, run `mochiflow freeze`, run `cargo test` | `engine/MANIFEST.json` updates, `contracts.lock` hash unchanged, version gate does not trip, `doctor engine` clean |

> Automated freeze write/check tests run against a `tempfile` repo fixture and
> must not mutate the real repo tree (`engine/VERSION` / `contracts.lock` /
> `engine/MANIFEST.json`). The manual QA rows above that imply mutation
> (QA-02 version bump, QA-04 hand-edit) are performed on a throwaway checkout or
> reverted afterward; the only real-tree check is the read-only
> `mochiflow freeze --check`.
