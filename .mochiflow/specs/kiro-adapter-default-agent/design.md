# Kiro adapter: drop dedicated agent, single always-on steering, delegate permissions â€” Design

## Design Decisions

- **Two-file generated surface.** Replace the dedicated `spec-builder.json` and
  the eight `spec*.md` verb steering files with one always-on steering file
  (`.kiro/steering/mochiflow.md`) plus the unchanged read-only reviewer agent
  (`.kiro/agents/spec-independent-reviewer.json`). Rationale: parity with the
  `AGENTS.md` / `CLAUDE.md` prose-enforcement model, minimal generated surface,
  and minimal `engine/MANIFEST.json` churn. Rejected: keeping a minimal agent
  only to wire reviewer-subagent trust â€” the host default agent can invoke the
  reviewer with an accepted per-call allow.
- **`#[[file:...]]` pointers, not inlined context.** `mochiflow.md` includes the
  router, both constitution files, and the three context files by Kiro-native
  reference. Rejected inlining: it would re-render `mochiflow.md` on every
  `refresh-context` and bloat engine integrity hashing, buying no real
  certainty over the pointer mechanism already used for the router. Primary
  source: Kiro steering `#[[file:...]]` file-reference semantics
  (Kiro docs, current `.kiro/steering` behavior).
- **No baked `toolsSettings`.** All shell/write/web_fetch policy is removed from
  generated Kiro JSON; guardrails live in `mochiflow.md` Rules prose. Permissions
  are owned by the user's `permissions.yaml` (per-user, outside the repo, so not
  vendorable). The `git push` / provider-PR invariant becomes prose, matching
  Claude/AGENTS.
- **Self-healing regeneration (alpha, breaking-OK).** Rather than a migration
  command, `adapter generate` carries a static list of deprecated Kiro output
  paths and removes those that contain the `generated by mochiflow` marker,
  reporting each. Marker-less (user-edited / hand-authored) files are skipped.
  Rationale: the marker is already the system's reliable "mochiflow-managed"
  signal (used by `detach`); this self-heals existing projects (including this
  dogfood repo) without risking user files such as `.kiro/steering/release.md`.
- **No `VERSION` / `contracts.lock` bump.** The changed contract is the
  `engine/`-resident adapter output (hashed into `engine/MANIFEST.json`), not a
  `contracts/*.json` schema. The dogfood sequence (`mochiflow freeze` â†’
  `mochiflow upgrade --source engine` â†’ `mochiflow adapter generate`) restores
  integrity; `doctor` / `freeze --check` confirm. A `VERSION` bump is performed
  only if `doctor`/`freeze` reveal a schema-surface change (not expected).

## Data Model / Interfaces

- `AdapterResult` gains a way to report removed deprecated files (e.g. a
  `removed: Vec<String>`), printed by `cmd_adapter_generate` alongside `wrote:`.
  In `--check` mode, lingering marker-bearing deprecated files are reported as
  drift so CI catches un-regenerated projects.
- Deprecated Kiro path list (static, kiro-only):
  `.kiro/agents/spec-builder.json`, `.kiro/steering/spec.md`,
  `.kiro/steering/spec-discuss.md`, `-plan`, `-build`, `-ship`, `-patch`,
  `-review`, `-refresh-context`.
- Marker check reuses `MARKER_PREFIX` (`generated by mochiflow`).

## Error Handling

- Removal that hits `NotFound` is a no-op success (file already gone).
- A deprecated path present but lacking the marker is skipped and reported as
  preserved (never deleted).
- Self-heal applies only to the `kiro` adapter; other adapters are unaffected.

## Test Strategy

- Update `conformance.rs` `materialize_full` assertions: expect
  `.kiro/steering/mochiflow.md` (not `spec.md`) and
  `.kiro/agents/spec-independent-reviewer.json`; drop `spec-builder.json`

- Update `cli.rs` cases that write/inspect `spec-builder.json`.
- Update `present.rs` doctests/tests referencing `spec-builder.json` messages.
- New unit test in `adapter.rs`: deprecated markered files removed + reported;
  marker-less `release.md`-style file preserved; reviewer agent still generated.
- Dogfood integrity: after edits run `mochiflow freeze`,
  `mochiflow upgrade --source engine`, `mochiflow adapter generate`, then
  `mochiflow doctor` and `cargo test`.

## Integration Contract

- Contract owner: the `kiro` adapter generator (mochiflow), consumed by Kiro CLI.
- Output contract (new): `.kiro/steering/mochiflow.md` (`inclusion: always`,
  `#[[file:]]` includes + Rules) and `.kiro/agents/spec-independent-reviewer.json`
  (read-only). No other mochiflow-managed files under `.kiro/`.
- Compatibility: breaking for the prior Kiro layout (alpha, accepted). Existing
  projects are converged by self-healing regeneration, not a migration command.
- Permissions: delegated entirely to the user's `permissions.yaml`; mochiflow
  emits nothing for it. Guardrails are advisory prose in `mochiflow.md`.
- Failure handling: if generation cannot write the new files, existing behavior
  (error reporting in `AdapterResult.errors`) applies; deprecated files are only
  removed after their marker is confirmed.
- Verification: `mochiflow adapter generate --check` + `mochiflow doctor` +
  `cargo test`.
