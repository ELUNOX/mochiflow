# Support local-only MochiFlow specs when .mochiflow is gitignored

## Problem

MochiFlow currently leans on tracked spec artifacts during PR delivery. The
mechanical `accept` close-out stages `.mochiflow/specs/{slug}/` and linked ADR
records, and `mochiflow pr` requires an accepted spec commit with a `Spec:`
trailer before it will push and create the PR.

That is correct for repositories that track `.mochiflow/`, but it fails in
repositories that intentionally gitignore `.mochiflow/`. In those repositories,
forcing `git add -f .mochiflow/...` would violate the repository's persistence
policy. The workflow needs to distinguish tracked specs from local-only specs
explicitly while preserving verification, review, and PR handoff quality.

## Appetite

This is worth a design-level CLI change, not a small patch. It crosses the
`accept`, `pr`, open-procedure, docs, and integration-test surfaces, but should
not redesign the full spec lifecycle or introduce a new persistence backend.

The target is a conservative v1 that keeps tracked mode unchanged and adds a
clear local mode for ignored `.mochiflow/` repositories.

## Solution

Add a shared spec persistence mode detector in the core CLI and use it from
`accept`, `pr`, and the engine open guidance.

Mode detection should use Git ignore behavior as the source of truth:

- **Tracked mode** when the concrete spec artifact path is not ignored.
- **Local mode** when `.mochiflow/` or the configured specs directory is ignored
  such that the spec artifacts are local-only.

Do not add a config override in the first version. The repository's `.gitignore`
policy is already the user-authored declaration of whether `.mochiflow/`
artifacts are meant to be committed.

In tracked mode, preserve the existing behavior:

- `mochiflow accept` runs final verification, updates evidence and metadata,
  stages the spec and linked ADR records, and creates the accepted close-out
  commit with a `Spec:` trailer.
- `mochiflow pr` requires the committed accepted spec and `Spec:` trailer before
  it pushes or dispatches a backend.

In local mode, change `accept` to preserve the quality gates without attempting
to commit local-only artifacts:

- run final verification for declared surfaces;
- validate lint, AC Matrix completeness, and required review result;
- update local `spec.md` verification evidence and `spec.yaml` to
  `status: accepted`;
- skip close-out commit, spec staging, and ADR staging;
- print an explicit success message that names local mode and the skipped git
  actions;
- never suggest `git add -f` for `.mochiflow/...`.

In local mode, change `pr` preflight so it no longer requires a committed
accepted spec or `Spec:` trailer. Instead it should require:

- clean tracked working tree;
- current branch is the source branch;
- base and head are different;
- head is ahead of base;
- local spec state is `accepted`;
- verification evidence and review result are complete enough for acceptance.

The PR body generated by the open flow should carry the reviewable evidence that
would otherwise have been reviewed through committed spec artifacts:

- final verification evidence;
- review result;
- a durable decision summary derived from the local fold/ADR material or the
  close-out notes prepared by open.

Update engine guidance and user docs so `open` does not direct local-mode users
to force-add ignored `.mochiflow/` paths. Documentation should describe tracked
mode and local mode, the mode detection rule, constraints, recommended use, and
the migration path from local mode to tracked mode.

## Rabbit Holes

- Do not implement a general persistence backend or remote spec store.
- Do not weaken tracked mode's accepted close-out commit or `Spec:` trailer
  preflight.
- Do not make `git add -f` the documented workaround for local mode.
- Do not treat `mochiflow pr --dry-run` as a complete behavioral substitute for
  the real preflight unless the command's dry-run contract is intentionally
  changed and tested.
- Do not move PR body generation into `mochiflow pr`; `open` remains the owner
  of external-reviewer-facing content.

## No-gos

- No forced tracking of `.mochiflow/` artifacts in local-mode repositories.
- No committed accepted spec requirement in local-mode PR preflight.
- No config migration that changes existing tracked repositories into local mode.
- No schema-breaking change to `contracts/pr-request.schema.json`.
- No `status: done`, `_done/` moves, or committed `INDEX.md` behavior changes.

## Alternatives Considered

- **Require `git add -f .mochiflow/...` in ignored repositories.** Rejected
  because it contradicts the repository's ignore policy and makes local-only
  specs an accidental unsupported mode.
- **Add `specs.mode = "tracked" | "local"` immediately.** Rejected for v1
  because it creates a second source of truth that can disagree with
  `.gitignore`. A future override can be added if real repositories need it.
- **Drop the accepted state requirement for local mode.** Rejected because local
  mode should change persistence, not quality gates.
- **Keep PR preflight unchanged and only document local mode as unsupported.**
  Rejected because the requested workflow is valid: implementation commits are
  tracked and reviewable, while spec artifacts are intentionally local.
- **Fold local-mode decisions only into ADR files.** Rejected as insufficient
  because local ADR files are not reviewable in the PR; the PR body must carry a
  reviewer-facing decision summary.
